# 📊 ИТОГОВЫЙ АНАЛИЗ: RTSP vs WebSocket для HeroSpeed NVR

## 🔍 Цель исследования

Определить возможность использования стандартного RTSP протокола (порт 554) вместо proprietary WebSocket API (порт 80) для получения видеопотока с HeroSpeed/Longse NVR.

---

## 📋 Методология

1. **Анализ PCAP дампа** (`vms_capture.pcap`)
   - Объем: 11,035 пакетов, 7.6 MB
   - Длительность: 40.7 секунд
   - Инструменты: tshark, strings, grep

2. **Практическое тестирование**
   - ffprobe (ffmpeg)
   - VLC (cvlc)
   - Python subprocess

3. **Проверенные URL форматы**
   - Из дампа: `/1`, `/11`, `/21`, `/31`
   - От пользователя: `/00`, `/01`
   - Гипотетические: `/2`, `/3`, `/10`, `/20`, `/30`

---

## ✅ Найденные факты

### 1. **Структура RTSP сеанса (из дампа)**

```
DESCRIBE rtsp://10.120.204.4:554/11
  ↓
200 OK + Content-Base: rtsp://10.120.204.4/1
  ↓
SETUP rtsp://10.120.204.4/1/trackID=0 (видео)
  ↓
SETUP rtsp://10.120.204.4/1/trackID=1 (аудио)
  ↓
PLAY rtsp://10.120.204.4/1
  ↓
RTP данные (H.264 + PCMA)
```

### 2. **Авторизация**

- Метод: **Digest Authentication**
- Realm: `"Surveillance Server"`
- **КРИТИЧНО:** Разные URI для разных команд:
  - DESCRIBE: `uri="rtsp://10.120.204.4:554/11"` (с портом)
  - SETUP/PLAY: `uri="rtsp://10.120.204.4/1"` (без порта!)

### 3. **Кодеки**

- Видео: H.264, RTP payload 96, 90kHz
- Аудио: PCMA (G.711), RTP payload 8, 8kHz
- Транспорт: TCP (interleaved)

### 4. **Клиент в дампе**

- Библиотека: **LIVE555 Streaming Media v2015.01.19**
- Это C++ библиотека с полной поддержкой RTSP RFC

## ❌ Проблемы с RTSP

### **Проблема 1: Нестандартное поведение сервера**

```
Content-Base: rtsp://10.120.204.4/1  ← БЕЗ порта :554
```

Большинство RTSP клиентов:
- Игнорируют Content-Base
- Используют оригинальный URL для всех команд
- Не меняют порт между командами

**Результат:** Авторизация не проходит (неверный URI для Digest hash)

---

### **Проблема 2: Специфичная реализация**

Сервер требует **строгой последовательности**:
1. DESCRIBE с одним URI
2. Два SETUP с другим URI (из Content-Base)
3. PLAY с тем же URI
4. Session ID должен сохраняться

Стандартные клиенты (ffmpeg, VLC):
- Упрощают RTSP handshake
- Могут пропускать шаги
- Не всегда корректно обрабатывают Content-Base

**Результат:** Таймаут подключения

---

### **Проблема 3: Практические тесты**

| Тест | Результат | Детали |
|------|-----------|--------|
| ffprobe (первый запуск) | ⚠️ Частичный успех | Показал кодеки, но завис |
| ffprobe (повторные) | ❌ Провал | Таймаут или "Invalid data" |
| ffmpeg (детальный) | ❌ Провал | Таймаут подключения |
| VLC (cvlc) | ❌ Провал | Таймаут подключения |
| Каналы 1-16 массово | ❌ Провал | Ни один не работает |

**Вывод:** Первый "успех" ffprobe был ложным - он получил SDP описание, но не смог получить видеоданные.

## ✅ WebSocket API (текущее решение)

### **Преимущества:**

1. **Стабильность:**
   - ✅ Уже работает в продакшене
   - ✅ Поддерживает общую сессию для всех камер
   - ✅ Автоматическое переподключение

2. **Функциональность:**
   - ✅ Получение видео (substream)
   - ✅ Управление PTZ
   - ✅ Настройки камер
   - ✅ Статус системы

3. **Интеграция:**
   - ✅ Полный контроль над сессиями
   - ✅ Обработка ошибок
   - ✅ Логирование

### **Недостатки:**

1. ❌ Proprietary протокол
2. ❌ Сложнее код (~100 строк vs ~20 для RTSP)
3. ❌ Зависимость от vendor API
4. ❌ Требуется авторизация через HTTP перед WebSocket

---

## 🎯 ИТОГОВАЯ РЕКОМЕНДАЦИЯ

### **❌ НЕ ПЕРЕХОДИТЬ НА RTSP**

**Причины:**

1. **Несовместимость со стандартными клиентами**
   - ffmpeg/VLC не работают
   - Требуется кастомная реализация RTSP клиента
   - Поддержка Content-Base логики

2. **Нет преимуществ перед WebSocket**
   - RTSP не проще в реализации (учитывая специфику сервера)
   - Нет улучшения производительности
   - Потеря функциональности (PTZ, настройки)

3. **Риски стабильности**
   - Текущее WebSocket решение работает стабильно
   - RTSP потребует месяцев отладки
   - Нет гарантии работы с разными версиями прошивки NVR

- Нестандартной реализации сервера
- Сложностей с Content-Base логикой
- Несовместимости со стандартными клиентами
- Отсутствия реальных преимуществ перед WebSocket
