# Использование RTSP (порт 554) вместо WebSocket API

## 📋 Обзор

Текущая реализация использует **WebSocket API** HeroSpeed NVR для получения видеопотока. Альтернативный подход - использование стандартного протокола **RTSP (Real Time Streaming Protocol)** на порту 554.

---

## 🔍 Сравнение подходов

### Текущий подход: WebSocket API (порт 80)

**Как работает:**
```python
1. Авторизация через HTTP API → получаем sessionID
2. Подключение к ws://host:80/
3. Отправка JSON команды play с sessionID
4. Получение бинарных пакетов H.264/H.265
5. Декодирование через ffmpeg
```

**Преимущества:**
- ✅ Интегрировано с системой аутентификации HeroSpeed
- ✅ Поддержка управления (PTZ, настройки)
- ✅ Единый порт для API и видео
- ✅ Работает стабильно с текущей прошивкой

**Недостатки:**
- ❌ Нестандартный протокол (vendor-specific)
- ❌ Требует сложной авторизации
- ❌ Зависимость от конкретной реализации API
- ❌ Меньше документации и инструментов

---

### Альтернатива: RTSP (порт 554)

**Как работает:**
```python
1. Прямое подключение к rtsp://user:pass@host:554/...
2. Стандартный handshake RTSP (OPTIONS, DESCRIBE, SETUP, PLAY)
3. Получение RTP пакетов с видео
4. Декодирование через ffmpeg
```

**Преимущества:**
- ✅ **Стандартный протокол** (RFC 2326, RFC 7826)
- ✅ **Широкая поддержка** (VLC, ffmpeg, OpenCV, GStreamer)
- ✅ **Проще интеграция** (не нужна сложная авторизация)
- ✅ **Лучшая документация** и инструменты отладки
- ✅ **Совместимость** с разными производителями
- ✅ **Меньше кода** (нет необходимости в WebSocket клиенте)

**Недостатки:**
- ⚠️ Может требовать отдельной настройки на NVR
- ⚠️ Разные производители используют разные URL форматы
- ⚠️ Может не поддерживать некоторые функции (PTZ через RTSP ограничен)
- ⚠️ Порт 554 должен быть открыт

## 🔄 Интеграция RTSP в cctv_stream.py

Если RTSP работает, можно заменить WebSocket на RTSP. Вот пример модификации:

### Вариант A: Полная замена WebSocket на RTSP

```python
def _rtsp_feed(cam, proc_holder, scale, q, fps):
    """Получение видеопотока через RTSP вместо WebSocket"""
    channel = cam["channel"]
    nvr_host = cam["nvr_host"]
    nvr_user = cam["nvr_user"]
    nvr_pass = cam["nvr_pass"]
    
    # Формируем RTSP URL (субпоток для thumbnails)
    rtsp_url = f"rtsp://{nvr_user}:{nvr_pass}@{nvr_host}:554/cam/realmonitor?channel={channel+1}&subtype=1"
    
    # Запускаем ffmpeg для получения RTSP потока
    cmd = [
        "ffmpeg",
        "-rtsp_transport", "tcp",  # Используем TCP для надежности
        "-i", rtsp_url,
        "-vf", f"scale={scale}",
        "-q:v", str(q),
        "-r", str(fps),
        "-f", "mjpeg",
        "pipe:1"
    ]
    
    proc = subprocess.Popen(
        cmd,
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
        bufsize=10**8
    )
    
    proc_holder[0] = proc
    store = proc_holder[1]
    cam_id = proc_holder[2]
    
    with proc_lock:
        sub_processes[cam_id] = proc
    
    # Читаем кадры из stdout ffmpeg
    _read_frames(proc, store, cam_id)
```

## 📊 Преимущества перехода на RTSP

### 1. Упрощение кода

**До (WebSocket):**
- ~100 строк кода для WebSocket клиента
- Сложная обработка бинарных пакетов
- Ручное определение кодека
- Управление сессиями через HTTP API

**После (RTSP):**
- ~20 строк кода для запуска ffmpeg
- Стандартная обработка потока
- Автоматическое определение кодека
- Нет необходимости в сессиях для видео

### 2. Надежность

**WebSocket API:**
- Зависит от конкретной реализации HeroSpeed
- Может ломаться при обновлении firmware
- Требует поддержания сессии

**RTSP:**
- Стандартный протокол (RFC)
- Работает с любым NVR
- Не требует сессий (аутентификация в URL)

### 3. Производительность

**WebSocket:**
- Дополнительная нагрузка на CPU (обработка JSON, WebSocket)
- Задержки из-за обработки пакетов

**RTSP:**
- Прямая передача RTP пакетов
- Минимальные задержки
- Оптимизировано в ffmpeg

## ⚙️ Настройка NVR для RTSP

### Через веб-интерфейс:

1. Войдите в веб-интерфейс NVR
2. Перейдите: **Configuration → Network → Advanced Settings → RTSP**
3. Убедитесь, что RTSP включен
4. Проверьте порт (должен быть 554)
5. Сохраните настройки

### Через ONVIF:

Если NVR поддерживает ONVIF, можно использовать ONVIF Device Manager для:
- Поиска правильных RTSP URL
- Проверки настроек
- Тестирования потоков

## 🎯 Рекомендации

### Когда использовать RTSP:

✅ **Рекомендуется RTSP, если:**
- Порт 554 открыт и доступен
- RTSP успешно проходит тестирование
- Нужна максимальная совместимость
- Хотите упростить код
- Планируете поддержку разных производителей NVR

❌ **Оставайтесь на WebSocket, если:**
- RTSP отключен или не работает
- Нужны расширенные функции управления (PTZ)
- Текущая реализация стабильна
- Нет времени на тестирование и миграцию

## 📝 Примечания

### Безопасность RTSP

⚠️ **Важно:** RTSP URL содержит логин и пароль в открытом виде!

**Меры безопасности:**
1. Не сохраняйте URL в логах
2. Используйте отдельные учетные записи для RTSP
3. Ограничьте доступ к порту 554 фаерволом
4. Рассмотрите использование RTSP over TLS (rtsps://)

### Кодеки

HeroSpeed NVR обычно поддерживает:
- **H.264** (AVC) - наиболее совместимый
- **H.265** (HEVC) - лучшее сжатие, но требует больше ресурсов

RTSP автоматически определяет кодек, в отличие от WebSocket где нужно ручное определение.
